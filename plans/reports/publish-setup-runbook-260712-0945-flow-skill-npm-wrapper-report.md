## Critical first-publish caveat (from research 2026-07)

**The very first publish of `@manhquy/flow-skill` cannot use Trusted Publisher.** npm requires the package to already exist on the registry before Trusted Publisher can be configured against it. There are two paths:

- **Path A — token-first, then Trusted Publisher (recommended)**:
  1. Manually publish `v0.1.0-rc.1` from your local machine with a classic npm token (`npm login` + `npm publish --access public --tag rc`).
  2. On the npm dashboard, configure Trusted Publisher for the now-existing package.
  3. From `v0.1.0-rc.2` onwards, the workflow does everything via OIDC — no more tokens on your laptop.

- **Path B — setup-npm-trusted-publish tool**: creates an empty placeholder version so Trusted Publisher can bind. Less battle-tested; skip unless you have a reason.

The runbook below covers Path A. Path B is a future exercise.

### Step 4a. FIRST PUBLISH ONLY — manual with npm token, NO provenance

Trusted Publisher cannot bind to a package that does not exist yet. Publish `v0.1.0-rc.1` from your local machine once.

> **Important**: `npm publish --provenance` **does not work from a laptop** — provenance is generated by npm only inside a supported CI environment (GitHub Actions / GitLab CI / CircleCI Cloud). The command errors with "unsupported CI environment". So `rc.1` will not carry a provenance attestation. `rc.2` onward (via the workflow) will.

```bash
cd D:/project/flow/flow-skill/npm-wrapper
npm run sync                                # materialize skills/flow from ../skills/flow
npm test                                    # 35/35 must pass
npm pack --dry-run --ignore-scripts --json  # audit; unpacked size < 1MB

# Login writes a bearer token to ~/.npmrc in plaintext. Do NOT run this from a directory
# where npm might create a project-scoped ./.npmrc — cd out of the repo first if you're
# nervous. Verify the token file location before continuing.
npm login                                   # prompts email + password + 2FA OTP
ls -la ~/.npmrc                             # confirm the token lives here, not in the repo
git status --porcelain | grep -iE '\.npmrc|\.tgz' && \
  echo 'FAIL: npm artifact detected in working tree — remove before proceeding' || \
  echo 'clean'

# NOTE: no --provenance flag on this first publish (see above).
npm publish --access public --tag rc

# Verify the release exists and inspect the dist-tag surface. Expect only `rc` at this
# point — no `latest` — see Path A note below about install semantics.
npm view @manhquy/flow-skill@0.1.0-rc.1 version
npm view @manhquy/flow-skill dist-tags
```

**Immediately revoke the token** (deleting `~/.npmrc` alone leaves the token live server-side):

```bash
npm logout               # closes the local session
npm token list           # find the id for the token just used
npm token revoke <id>    # kill it on the registry so a leaked ~/.npmrc is useless
rm -f ~/.npmrc           # optional; the token is already dead
```

Path A install-semantics warning: with only `rc: 0.1.0-rc.1` set, a user running `npm install @manhquy/flow-skill` **without** an explicit tag will still resolve to the RC (npm falls back to the highest published version when `latest` is absent). This is documented in the RC-window README, but if you want the "no pin = nothing" behavior, publish an empty placeholder later with `--tag latest` (out of scope for this runbook).

Every subsequent publish is via the workflow (Steps 6–8) and does carry provenance.
