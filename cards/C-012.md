# C-012 — honest cycle accounting (build-intent vs diagnostic, read-time)

status: done
deps: C-011
group: telemetry-truth
implements: W2

## Scope

The "cycles started / reached-cards" metric is dishonest: diagnostic-only sessions (`status`,
`doctor`, `usage`, `recall`) get a `cycle_id` from the universal `_ensure_cycle` (`flow.sh:1401`),
so a user who only ran `status` looks like an abandoned build. Evidence: 7/7 "abandoned" device-wide
cycles were pure diagnostic sessions with zero stage progression.

**Decision (operator-confirmed, red-team-hardened): fix at READ time, do NOT touch logging.**
v0.11 FR2 deliberately stamps a `cycle_id` at every entry point so brownfield/`assess` is not blind —
that decision is verified and stays. We therefore must NOT gate `_ensure_cycle` (that would reverse
FR2, erase killed-at-gate + `assess` cycles, and break reproducibility of the historical log).

Instead, the rollup (`flow_harness.py`) classifies each cycle using the **already-logged `read_only`
event field**: a cycle is a **build cycle** iff it has ≥1 non-`read_only` event (`next`/`card`/`skip`/
`auto`/`assess`/stage-unlock). `usage` then reports BOTH, additively — the old total is kept, the
honest build-intent count is surfaced beside it:

```
cycles started:        12   (build-intent: 5 · diagnostic-only: 7)
cycles reached cards:  5
```

This is **retroactive** (fixes the existing 1739-line log's reported number with no rewrite),
**reversible** (pure read-time classification), and **FR2-preserving** (every cycle is still
stamped and logged). An optional `--builds-only` view filters to build cycles. Killed-at-gate
cycles (which ran `next`) and `assess` cycles (non-`read_only`) correctly remain build cycles.

Legacy rows missing the `read_only` field: classify by command name via the same allowlist the
runner uses (`_log_is_readonly`) as a COALESCE fallback, so old logs classify too.
