## Scope

The Review gate (`adversarial-review.md:6`) uses a generic `code-reviewer` lens. The skill already
ISOLATES security-class cards: a security-class card review is one of the named triggers for the
optional Codex 4th lens (`adversarial-review.md:54`) and the auto-run Tier-C HALT. But at that
existing trigger the *primary* lens is still the generic reviewer — there is no security-specialist
lens. Fix: when a card is security-class (the trigger already defined), the Review gate runs the
`security-reviewer` agent (STRIDE/OWASP, secrets, injection, SSRF — registry-present) as the
acceptance lens, layered with `code-reviewer`, and (when USABLE) the cross-model Codex lens stays
as the cross-vendor check. Non-security cards are unchanged (generic `code-reviewer`). Markdown
layer only.

Gate parity (PRD §NFR): the security lens INFORMS triage; the same Review gate still decides — it
never auto-fails or lowers the gate. No-defang rule (`adversarial-review.md:8-23`) applies to the
security dispatch too: diff + contract + acceptance, no "what not to find" steer.

**Critical — the lens MUST NOT release the Tier-C HALT.** The security-class operator HALT
(`auto-run.md:13,57`) is triggered by CLASSIFICATION and resolved ONLY by written operator
acknowledgment in `DEBT.md`. A `security-reviewer` "looks fine" verdict is advisory evidence for
the operator — it can NEVER substitute for, or auto-release, the HALT. State this explicitly.

**Portability / agent-vs-skill:** the delegated lens is the `security-reviewer` AGENT (invoked via
`Task(subagent_type=...)`). `ck-security` is a SKILL (main-context Skill tool, no subagent
isolation) — it may be named as an optional inline orchestrator pass, never as the delegated
subagent. Detect-first degrade: `security-reviewer` absent -> `code-reviewer` runs an explicit
STRIDE/OWASP checklist -> inline security review against the Tier-C keyword list. Gate identical
on every rung.
