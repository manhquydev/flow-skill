# C-015 — engine-contract hardening: atomic lock + honest _python exit

status: done

group: engine-hardening

## Scope

Two tightly-cohesive engine-contract fixes in `flow.sh`, bundled because both are
"the helper must not lie about its result" and both are small disjoint functions:

**W5 — `lock_acquire` TOCTOU.** `lock_acquire` (`flow.sh:274-308`) reads the lock (`_read_lock`,
~282) then later writes it (`_write_lock`, ~308) with a non-atomic gap: two sessions can both pass
the read checks and both write the lock. v0.11 FR4 made the lock hard-BLOCK and added dead-process
reclaim, but the CLAIM itself is still check-then-write. Fix: add an atomic `mkdir` **GUARD
AROUND** `_write_lock` — NOT a replacement. `mkdir "$LOCK_DIR"` is the POSIX-atomic test-and-set
that exactly one racer wins; the winner then writes the **existing FR4 metadata file**
(`TS|OWNER|PID|HOST|CMD`) so dead-PID `kill -0` reclaim, TTL stale reclaim, FLOW_FORCE takeover,
same-session refresh and weak-owner warn ALL keep working unchanged. The loser falls through to the
existing BLOCK/reclaim path. A legacy flat lock file (no dir) is still tolerated so an in-flight
upgrade doesn't strand a lock.

**Windows gate (operator-mandated): `mkdir` atomicity is NOT assumed on Git-Bash/Windows.** A
2-session parallel race test MUST run and pass on the Windows/Git-Bash runner. **If atomicity does
NOT hold there, W5 auto-DEFERS** — revert to the v0.11 content-lock, ship C-015 with W6 only, and
record the deferral in `DEBT.md`. FR4 already covers the main failure (crash), so deferring is safe.

**W6 — `_python()` honest exit.** `_python` (`flow.sh:114`) does `return 0` after the loop falls
through with NO interpreter found and EMPTY stdout — exit 0 means "success" but nothing was found.
Callers currently guard on empty *value* (`[ -n "$py" ]`) so no live bug, but the contract is a
trap for future callers. Fix: `return 1` on the no-interpreter fall-through; keep `command -v "$p"; return 0`
on success. Audit the 3 call sites (`harness_call:121`, `harness_available:129`, `cmd_harness:142`,
`harness_emit:136` via `_python` inline) — they read the value, so the exit-code change is additive
and safe.
